WebSockets — Study Guide

Full-duplex real-time communication for the web

1. What are WebSockets?

WebSocket is a communication protocol (RFC 6455, 2011) that provides full-duplex (bidirectional) communication over a single, long-lived TCP connection. Unlike HTTP's request-response model, WebSockets allow both client and server to send messages independently at any time — without re-establishing a connection.

Key characteristics:

  • Full-duplex: client and server send/receive simultaneously
  • Single persistent TCP connection — established once, kept open
  • Low overhead: 2-14 byte frame header vs. hundreds of bytes in HTTP headers
  • Low latency: no connection setup per message
  • Works over standard ports: ws:// (port 80) and wss:// (port 443)
  • Supported in all modern browsers and most server runtimes

3. How WebSockets Work — The Handshake

A WebSocket connection starts as a standard HTTP request that is then "upgraded" to the WebSocket protocol. This is called the WebSocket handshake.

Step-by-step process:

  • 1. Client sends HTTP GET with Upgrade: websocket and Sec-WebSocket-Key headers
  • 2. Server responds with HTTP 101 Switching Protocols and Sec-WebSocket-Accept
  • 3. TCP connection is now a WebSocket — both sides can send frames freely
  • 4. Either side can close the connection by sending a Close frame

Client → Server: HTTP Upgrade Request

GET /chat HTTP/1.1
Host: example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Version: 13

5. Real Example — Chat Application

A complete real-time chat app. The Node.js server maintains a list of all connected clients and broadcasts every received message to all of them.

Node.js Server (npm install ws)

server.js (Node.js + ws library)

const WebSocket = require('ws');
const http = require('http');

const server = http.createServer();
const wss = new WebSocket.Server({ server });

wss.on('connection', function(ws, req) {
    const ip = req.socket.remoteAddress;
    console.log('Client connected:', ip);

    // Welcome the new client
    ws.send(JSON.stringify({ type: 'system', text: 'Welcome to the chat!' }));

    ws.on('message', function(data) {
        const msg = data.toString();
        console.log('Received:', msg);

        // Broadcast to ALL connected clients
        wss.clients.forEach(function(client) {
            if (client.readyState === WebSocket.OPEN) {
                client.send(msg);
            }
        });
    });

    ws.on('close', function(code, reason) {
        console.log('Client disconnected. Code:', code);
    });

    ws.on('error', function(err) {
        console.error('Socket error:', err.message);
    });
});

server.listen(8080, () => console.log('WS server on port 8080'));

    input.addEventListener('keydown', (e) => {
      if (e.key === 'Enter') sendMessage();
    });

      const li = document.createElement('li');
      li.textContent = text;
      list.appendChild(li);
      list.scrollTop = list.scrollHeight;
    }

6. Auto-Reconnect Pattern

WebSocket has no built-in reconnection. Implement it manually with exponential backoff — start with a short delay and double it on each failed attempt.

JavaScript — Exponential Backoff Reconnect

let socket;
let reconnectDelay = 1000; // start: 1 second
const MAX_DELAY = 30000;   // cap at 30 seconds

function connect() {
    socket = new WebSocket('ws://localhost:8080');

    socket.onopen = function() {
        console.log('Connected');
        reconnectDelay = 1000; // reset on success
    };

    socket.onmessage = function(event) {
        console.log('Message:', event.data);
    };

    socket.onclose = function(event) {
        if (!event.wasClean) {
            console.log('Connection lost. Reconnecting in', reconnectDelay, 'ms');
            setTimeout(connect, reconnectDelay);
            reconnectDelay = Math.min(reconnectDelay * 2, MAX_DELAY);
        }
    };
}

connect();

WebSocket Configuration

WebSocketConfig.java

import org.springframework.context.annotation.Configuration;
import org.springframework.web.socket.config.annotation.*;

@Configuration
@EnableWebSocket
public class WebSocketConfig implements WebSocketConfigurer {

    @Override
    public void registerWebSocketHandlers(WebSocketHandlerRegistry registry) {
        registry.addHandler(new ChatHandler(), "/chat")
                .setAllowedOrigins("*");
    }
}

Chat Handler

ChatHandler.java

import org.springframework.stereotype.Component;
import org.springframework.web.socket.*;
import org.springframework.web.socket.handler.TextWebSocketHandler;
import java.util.List;
import java.util.concurrent.CopyOnWriteArrayList;

@Component
public class ChatHandler extends TextWebSocketHandler {

    // Thread-safe list — multiple threads may connect simultaneously
    private final List<WebSocketSession> sessions = new CopyOnWriteArrayList<>();

    @Override
    public void afterConnectionEstablished(WebSocketSession session) throws Exception {
        sessions.add(session);
        session.sendMessage(new TextMessage("Welcome!"));
    }

    @Override
    protected void handleTextMessage(WebSocketSession session,
                                     TextMessage message) throws Exception {
        String payload = message.getPayload();
        // Broadcast to all open sessions
        for (WebSocketSession s : sessions) {
            if (s.isOpen()) {
                s.sendMessage(new TextMessage(payload));
            }
        }
    }

    @Override
    public void afterConnectionClosed(WebSocketSession session,
                                      CloseStatus status) {
        sessions.remove(session);
    }
}

9. WebSocket Frames (Wire Format)

After the handshake, data is sent as WebSocket frames. Each frame has a small header followed by the payload. Understanding frames helps in debugging and protocol implementation.

Field

Size

Description

FIN

1 bit

Final fragment flag (1 = last/only fragment)

RSV1-3

3 bits

Reserved for extensions (usually 0)

Opcode

4 bits

0x1=text, 0x2=binary, 0x8=close, 0x9=ping, 0xA=pong

MASK bit

1 bit

1 if payload is masked (client→server MUST mask)

Payload Length

7-71 bits

≤125: in 7 bits; 126: next 2 bytes; 127: next 8 bytes

Masking Key

0 or 4 bytes

Present only if MASK=1 (client-to-server frames)

Payload

Variable

Application data (XOR'd with masking key if masked)

Client→Server frames MUST be masked. Server→Client frames are NOT masked.

10. Security — WSS and Best Practices

Always use wss:// (WebSocket Secure) in production. WSS runs WebSocket over TLS — the same encryption as HTTPS. Plain ws:// transmits data unencrypted.

  • wss:// — encrypted (TLS/SSL), required in production
  • ws:// — unencrypted, only acceptable on localhost
  • Validate the Origin header server-side to prevent Cross-Site WebSocket Hijacking (CSWSH)
  • Authenticate before upgrading — pass JWT in query string or use cookies
  • Limit message size to prevent memory exhaustion attacks
  • Rate-limit messages per connection to prevent abuse
  • Use wss:// behind a reverse proxy (nginx/Apache) that terminates TLS

JavaScript — Origin Validation

// Node.js: validate Origin to prevent CSWSH
const wss = new WebSocket.Server({
    port: 443,
    verifyClient: function(info, callback) {
        const allowed = ['https://myapp.com', 'https://www.myapp.com'];
        const isAllowed = allowed.includes(info.origin);
        callback(isAllowed, 403, 'Forbidden');
    }
});

12. Scaling WebSocket Servers

WebSocket connections are stateful — each connection is pinned to a specific server instance. Horizontal scaling requires two things: sticky sessions and a shared message broker.

  • Sticky sessions (session affinity): load balancer always routes a client to the same server node
  • Pub/Sub broker: Redis Pub/Sub, Kafka, or RabbitMQ — any node can publish, all nodes receive and forward to their own clients
  • Socket.IO: popular Node.js library with built-in Redis adapter for scaling and auto-reconnect
  • Spring WebSocket + STOMP + SockJS + Redis: common enterprise Java stack for scalable WebSockets

Node.js — Socket.IO with Redis Adapter (horizontal scaling)

// Redis adapter for Socket.IO (scales across multiple Node.js processes)
const { createClient } = require('redis');
const { createAdapter } = require('@socket.io/redis-adapter');
const { Server } = require('socket.io');

const pubClient = createClient({ url: 'redis://localhost:6379' });
const subClient = pubClient.duplicate();

const io = new Server(httpServer);
io.adapter(createAdapter(pubClient, subClient));

// Now multiple server nodes share the same pub/sub channel
io.on('connection', (socket) => {
    socket.on('message', (data) => {
        io.emit('message', data); // broadcast across ALL nodes
    });
});

13. Key Interview Questions & Answers

Q: What is WebSocket and how does it differ from HTTP?

A: WebSocket is a full-duplex, stateful protocol over TCP. After an HTTP "upgrade" handshake, both client and server can send messages independently. HTTP is stateless request-response — the client must request; the server can only respond. WebSocket has much lower overhead per message (small frame header vs. full HTTP headers).

Q: How does the WebSocket handshake work?

A: The client sends HTTP GET with headers: Upgrade: websocket, Connection: Upgrade, and a Sec-WebSocket-Key (random base64). The server responds with HTTP 101 Switching Protocols and Sec-WebSocket-Accept (SHA-1 of key + GUID, base64 encoded). After that, the TCP socket is reused as a WebSocket connection.

Q: What is the difference between ws:// and wss://?

A: ws:// is plain WebSocket over unencrypted TCP. wss:// is WebSocket over TLS (like HTTPS). Always use wss:// in production to encrypt data and prevent MITM attacks.

Q: When would you use WebSocket vs Server-Sent Events (SSE)?

A: Use WebSocket when you need bidirectional communication (chat, gaming, collaborative editing). Use SSE when only the server needs to push to the client (news feed, notifications) — SSE uses plain HTTP, works through all proxies, and has built-in reconnection.

Q: How do you implement reconnection in WebSocket?

A: WebSocket has no built-in auto-reconnect. On the onclose event, use setTimeout with exponential backoff to call new WebSocket(...) again. Reset the delay on successful connection. The cap prevents flood-reconnects. Libraries like ReconnectingWebSocket handle this automatically.

Q: How do you scale WebSocket servers horizontally?

A: WebSocket connections are sticky to one server. For scaling: configure the load balancer for sticky sessions (IP hash or cookie), and use a pub/sub message broker (Redis, Kafka) so that all server nodes receive all messages and can forward them to their local clients.

Q: What are WebSocket frames and why must client frames be masked?

A: Frames are the transmission unit of the WebSocket protocol. Each frame has a small header (FIN, opcode, MASK bit, payload length, optional masking key) followed by the payload. Client-to-server frames must be masked (XOR with a random 4-byte key) as per RFC 6455, to prevent cache-poisoning attacks against intermediary HTTP caches.

Q: What is Cross-Site WebSocket Hijacking (CSWSH)?

A: CSWSH is an attack where a malicious website initiates a WebSocket connection to your server using the victim's cookies (similar to CSRF). Mitigation: validate the Origin header server-side and reject connections from unexpected origins. Unlike XHR, WebSockets are not protected by the same-origin policy by default.
