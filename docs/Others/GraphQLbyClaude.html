1. What is GraphQL?

GraphQL is an open-source data query and manipulation language for APIs, and a runtime for fulfilling those queries with your existing data. Developed internally by Facebook in 2012 and open-sourced in 2015, it gives clients the power to ask for exactly what they need — nothing more, nothing less.

Key facts:

  • Created by Facebook (Meta) in 2012, open-sourced in 2015
  • Now maintained by the GraphQL Foundation under the Linux Foundation
  • A query language for your API — not a database query language
  • Works over HTTP — typically a single POST endpoint (/graphql)
  • Language-agnostic: servers available in Node.js, Python, Java, Go, Ruby, and more

2. Core Problems GraphQL Solves

2.1 Over-fetching

With REST, GET /users might return id, name, email, address, birthday, and avatar — even if you only need name and email. GraphQL lets you request only the fields you actually need.

2.2 Under-fetching (N+1 Problem)

With REST, getting a user and their posts might require two requests: GET /users/1, then GET /users/1/posts. With GraphQL you fetch all related data in a single round-trip.

2.3 Rapid Product Iteration

Front-end teams can request new fields without waiting for back-end API changes. Schema introspection also provides a self-documented API at all times.

4. Core Concepts

4.1 Schema

The schema is the central contract of a GraphQL API written in Schema Definition Language (SDL). It defines all available types, queries, mutations, and subscriptions, plus the relationships between types.

type Query {
  user(id: ID!): User
  users: [User!]!
}

type User {
  id: ID!
  name: String!
  email: String!
  posts: [Post!]!
}

type Post {
  id: ID!
  title: String!
  body: String
  author: User!
}

4.2 Type System

GraphQL defines 6 named type categories:

Scalar Types (built-in primitives)

  • Int — 32-bit signed integer
  • Float — double-precision floating point
  • String — UTF-8 character sequence
  • Boolean — true or false
  • ID — unique identifier (serialized as String)

You can also define custom scalars: scalar Date | scalar JSON | scalar URL

Object Types

The most common type — groups named fields together:

type Product {
  id: ID!
  name: String!
  price: Float!
  inStock: Boolean!
}

Enum Types

A set of allowed string values:

enum Status {
  ACTIVE
  INACTIVE
  PENDING
}

Interface Types

An abstract type that concrete types must implement:

interface Node {
  id: ID!
}

type User implements Node {
  id: ID!
  name: String!
}

type Post implements Node {
  id: ID!
  title: String!
}

Union Types

A field that can return one of several object types (no shared fields required):

union SearchResult = User | Post | Product

type Query {
  search(term: String!): [SearchResult!]!
}

Input Object Types

Used to pass complex arguments into mutations and queries:

input CreateUserInput {
  name: String!
  email: String!
  password: String!
}

type Mutation {
  createUser(input: CreateUserInput!): User!
}

Non-Null and List Modifiers

  • String — nullable string (can be null)
  • String! — non-null string (never null)
  • [String] — nullable list of nullable strings
  • [String!]! — non-null list of non-null strings

5. Queries

Queries are read operations. The client specifies exactly what fields it wants.

5.1 Basic Query

# Request
query {
  user(id: "1") {
    name
    email
  }
}

# Response
{
  "data": {
    "user": {
      "name": "Alice",
      "email": "alice@example.com"
    }
  }
}

5.2 Arguments

Any field can accept arguments:

query {
  users(limit: 10, offset: 20) {
    id
    name
  }
}

5.3 Aliases

Rename fields in the response to avoid conflicts:

query {
  adminUser: user(id: "1") { name }
  guestUser: user(id: "2") { name }
}

5.4 Fragments

Reusable field selections:

fragment UserFields on User {
  id
  name
  email
}

query {
  user(id: "1") { ...UserFields }
}

5.5 Variables

Pass dynamic values safely — avoids string interpolation vulnerabilities:

# Query definition
query GetUser($userId: ID!) {
  user(id: $userId) {
    name
    email
  }
}

# Variables (sent as JSON alongside the query)
{ "userId": "42" }

5.6 Directives

Modify execution dynamically using built-in directives:

query GetUser($withEmail: Boolean!, $skipPosts: Boolean!) {
  user(id: "1") {
    name
    email   @include(if: $withEmail)
    posts   @skip(if: $skipPosts) {
      title
    }
  }
}
  • @include(if: Boolean) — include the field only if the argument is true
  • @skip(if: Boolean) — skip the field if the argument is true
  • @deprecated(reason: String) — marks a schema field as deprecated

6. Mutations

Mutations are write operations (create, update, delete). Unlike query fields which run in parallel, mutation fields execute serially to avoid race conditions.

6.1 Basic Mutation

mutation {
  createUser(input: {
    name: "Bob"
    email: "bob@example.com"
  }) {
    id
    name
    email
  }
}

6.2 Mutation with Variables

mutation CreateUser($input: CreateUserInput!) {
  createUser(input: $input) {
    id
    name
  }
}

# Variables
{
  "input": {
    "name": "Bob",
    "email": "bob@example.com"
  }
}

6.3 Update and Delete

mutation {
  updateUser(id: "1", input: { name: "Robert" }) {
    id
    name
  }
}

mutation {
  deleteUser(id: "1") {
    success
    message
  }
}

7. Subscriptions

Subscriptions enable real-time updates. The client opens a persistent WebSocket connection; the server pushes data whenever a specified event occurs.

subscription {
  messageAdded(channelId: "general") {
    id
    text
    author {
      name
    }
    createdAt
  }
}

How it works:

  • 1. Client sends subscription operation over WebSocket
  • 2. Server registers the subscription
  • 3. When the event fires, the server pushes the payload to all subscribed clients
  • 4. Client receives real-time updates without polling
  • Common libraries: graphql-ws, subscriptions-transport-ws

8. Resolvers

A resolver is a server-side function that returns data for a specific field. Every field in your schema has a resolver (default resolvers return the property matching the field name).

// Node.js / Apollo Server example
const resolvers = {
  Query: {
    user: async (parent, { id }, context) => {
      return await context.db.users.findById(id);
    },
    users: async (parent, args, context) => {
      return await context.db.users.findAll();
    },
  },
  Mutation: {
    createUser: async (parent, { input }, context) => {
      return await context.db.users.create(input);
    },
  },
  User: {
    // field-level resolver: runs for every User returned
    posts: async (parent, args, context) => {
      return await context.db.posts.findByUserId(parent.id);
    },
  },
};

Resolver arguments:

  • parent — result of the parent field resolver
  • args — arguments provided in the query/mutation
  • context — shared object (auth token, DB connection, DataLoaders)
  • info — field name, schema info, query AST (advanced use)

9. Introspection

GraphQL APIs are self-documenting. Any client can query the schema itself using introspection — this powers GraphiQL autocomplete and Apollo Studio documentation.

{
  __schema {
    types {
      name
      kind
    }
  }
}

# Query a specific type
{
  __type(name: "User") {
    fields {
      name
      type { name }
    }
  }
}

10. Error Handling

GraphQL always returns HTTP 200. Errors appear in the errors array alongside any partial data:

{
  "data": {
    "user": null
  },
  "errors": [
    {
      "message": "User not found",
      "locations": [{ "line": 2, "column": 3 }],
      "path": ["user"],
      "extensions": {
        "code": "NOT_FOUND"
      }
    }
  ]
}
  • data — partial data for successfully resolved fields
  • errors — array of errors with message, locations, path, and extensions
  • A request can return BOTH data and errors (partial success is valid)

11. Best Practices

11.1 Schema Design

  • Design your schema around client needs, not the database structure
  • Use descriptive camelCase field names
  • Use Input types for mutation arguments (not inline scalars)
  • Paginate large lists — prefer cursor-based (Relay Connections) over offset
  • Implement the Node interface for globally unique IDs
  • Mark fields non-null (!) when they always have a value

11.2 Performance

  • Use DataLoader to batch and cache DB calls — solves the N+1 query problem
  • Set query depth and complexity limits to prevent resource exhaustion
  • Use persisted queries in production to enable caching and reduce payload size
  • Cache at the resolver level (e.g., Redis) for expensive computations

11.3 Security

  • Disable or restrict introspection in production
  • Enforce query depth limits (typically max 5-7 levels)
  • Enforce query complexity limits
  • Validate and sanitize all input arguments
  • Handle authentication in context (middleware), not in individual resolvers
  • Use allow-lists of known queries (persisted queries) for public APIs

12. Popular Tools & Ecosystem

12.1 Server Libraries

  • Apollo Server (Node.js) — most popular, full-featured, great docs
  • GraphQL Yoga (Node.js) — lightweight, modern, Fetch API based
  • Strawberry (Python) — code-first, type-annotation driven
  • Ariadne (Python) — schema-first
  • graphql-java (Java) — official Java library
  • Juniper (Rust) — high-performance, macro-based
  • Hot Chocolate (.NET) — feature-rich C# server

12.2 Client Libraries

  • Apollo Client — most popular; integrates with React, Vue, Angular
  • Relay — Facebooks production client; highly optimized, opinionated
  • urql — lightweight and extensible
  • graphql-request — minimal; great for server-side or scripts

12.3 Developer Tools

  • GraphiQL — in-browser IDE; uses introspection for autocomplete
  • Apollo Studio — enterprise explorer, schema registry, metrics
  • Postman — supports GraphQL requests
  • GraphQL Inspector — schema diffing and breaking-change detection
  • GraphQL Code Generator — auto-generate TypeScript types from schema

13. Quick Reference Cheat Sheet

# QUERY (read)
query [OperationName]($var: Type) {
  fieldName(arg: $var) {
    subField
  }
}

# MUTATION (write)
mutation [OperationName]($input: InputType!) {
  mutationField(input: $input) {
    returnedField
  }
}

# SUBSCRIPTION (real-time)
subscription {
  eventName {
    field
  }
}

# FRAGMENT (reusable selection)
fragment FragName on TypeName {
  field1
  field2
}

# DIRECTIVES
@include(if: Boolean)        # include field if true
@skip(if: Boolean)           # skip field if true
@deprecated(reason: String)  # schema only

# TYPE MODIFIERS
String     # nullable
String!    # non-null
[String]   # nullable list
[String!]! # non-null list of non-null strings

# BUILT-IN SCALAR TYPES
Int | Float | String | Boolean | ID
